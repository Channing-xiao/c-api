# Research: ai-security 模块技术调研与决策

## 决策概述

本调研基于 `spec.md` 中的 Open Questions 和工程约束，确定 ai-security 模块的架构选型、集成方式与关键实现策略。

---

## 1. AI 检测引擎的模型来源

### 问题
AI 检测引擎默认使用哪个模型接口？是否需要单独配置渠道和 API Key？

### 决策
复用 `new-api` 内部已有的渠道（Channel）和模型配置能力，不引入独立的 AI 安全模型 API Key。

### 理由
- `new-api` 已经具备完整的渠道管理、模型分发、额度控制、失败重试机制。
- 引入独立 API Key 会增加配置复杂度和密钥管理成本。
- 通过内部渠道调用模型，可利用现有的模型缓存、速率限制和监控。

### 实现方式
- 在 `custom/ai-security/engine/ai.go` 中通过内部 HTTP 客户端调用 `/v1/chat/completions` 或等效接口。
- 默认使用一个可配置的模型名称（如 `gpt-4o-mini` 或用户配置的轻量级模型）。
- 在 `aisec_configs` 中存储 `ai_model_name` 和 `ai_enabled` 配置。
- AI 检测设置 3 秒超时，超时或失败时降级为本地规则检测。

### 替代方案
- **独立 API Key**：更灵活，但增加配置负担和安全风险。
- **本地 ONNX/TensorFlow 模型**：无外部依赖，但部署包体积大、模型更新困难。

## 2. 默认规则种子数据

### 问题
默认规则种子数据的具体内容是否需要按行业定制？

### 决策
提供通用基础规则集，覆盖常见合规和安全场景；规则种子支持后续扩展和行业定制。

### 默认规则分类

| 分组 | 规则示例 | 检测类型 |
|---|---|---|
| 基础安全策略 | 政治敏感、暴恐、毒品、赌博 | Keyword + Regex |
| 个人隐私信息 | 身份证号、手机号、银行卡号、邮箱 | Regex + NER |
| 企业机密 | 内部项目代号、未公开财务数据 | Keyword + Regex |
| 合规风险 | 医疗记录、未成年人保护 | Regex + AI |
| Prompt 防护 | 越狱提示、角色扮演注入、系统提示泄露 | Keyword + AI |

### 理由
- 通用规则足以让大多数用户在首次安装后立即获得基础防护。
- 分组设计允许管理员按行业场景扩展，无需修改代码。
- 规则使用唯一 `code` 字段，支持幂等初始化和增量补充。

### 替代方案
- **按行业预置多套种子**：更贴合垂直场景，但增加维护复杂度和安装包体积。

## 3. 命中日志保留时长

### 问题
命中日志的默认保留时长是多少天？

### 决策
默认保留 30 天，可通过配置调整；提供手动清理和自动清理能力。

### 理由
- 30 天是安全审计的常见窗口期，满足多数合规需求。
- 日志量随请求量增长，无限保留会导致存储膨胀。
- 保留时长可配置，满足更严格或更宽松的策略。

### 实现方式
- `aisec_configs` 中存储 `log_retention_days`，默认 30。
- 提供 `/api/ai-security/configs` 接口修改。
- 每日定时任务清理过期日志，或利用数据库 TTL（如 ClickHouse 场景）。

## 4. 与官方 new-api 的集成方式

### 决策
采用"最小挂载点"策略，主项目只做 4 处必要改动：

| 挂载点 | 改动内容 | 文件 |
|---|---|---|
| 后端路由 | 注册 `/api/ai-security/*` | `router/api-router.go` |
| 初始化 | 调用 `ai_security.Init()` | `main.go` 的 `InitResources()` 或 `router.SetRouter()` |
| 请求检测 | 在 relay 请求前调用 `ai_security.CheckRequest()` | `router/relay-router.go` |
| 响应检测 | 在 relay 响应前调用 `ai_security.CheckResponse()` | `router/relay-router.go` |

### 理由
- 最小化与官方代码的耦合，降低后续升级冲突。
- 所有业务逻辑封装在 `custom/ai-security/` 包内，便于独立维护和卸载。

## 5. 数据库与迁移策略

### 决策
- 使用同一个数据库实例，但表名全部以 `aisec_` 前缀隔离。
- 迁移脚本放在 `custom/ai-security/migration/`，由模块内部 `Init()` 自动执行。
- 不修改 `model/main.go` 的官方迁移逻辑。

### 理由
- 独立数据库实例会增加部署复杂度（额外 DSN、连接池、备份）。
- 表前缀隔离已能满足"数据独立保存"的设计目标。
- 模块自管理迁移避免污染官方迁移流程。

---

## 6. 缓存策略

### 决策
- 使用内存缓存 + Redis（如果启用）双重缓存。
- 缓存键统一使用 `aisec:` 前缀。
- 规则和策略缓存 TTL 为 5 分钟；配置变更时主动失效缓存。

### 理由
- 安全检测在请求热路径上，缓存可减少数据库查询。
- 5 分钟 TTL 在一致性和性能之间取得平衡。
- 前缀隔离避免与官方缓存键冲突。

---

## 7. 前端集成策略

### 决策
- 前端页面、组件、API 请求全部放在 `custom/ai-security/web/`。
- 主项目只注册 `/ai-security` 路由和在 `System Settings > Security` 下增加入口。
- 菜单入口不显示为独立一级菜单，而是作为 `system-settings/security` 的"高级安全"子项。

### 理由
- 符合"产品体验上属于 sensitive-words 高级功能"的设计定位。
- 减少官方前端文件的改动范围。

---

## 8. 开关与降级策略

### 决策
- ai-security 使用独立配置开关 `ai_security_enabled`，存储在 `aisec_configs` 表中。
- 关闭 ai-security 时，中间件直接 `c.Next()`，不影响官方 sensitive-words。
- 官方 sensitive-words 的开关状态不影响 ai-security。
- AI 引擎超时或失败时降级为 Keyword + Regex + NER 本地检测。

### 理由
- 独立开关是两个功能解耦运行的关键。
- 降级策略保证高级检测不会导致请求长时间阻塞或失败。

---

## 9. 审计日志

### 决策
- 记录管理员对规则、分组、策略、配置的增删改操作。
- 审计日志单独存储在 `aisec_audit_logs` 表中。
- 审计日志默认保留 90 天。

### 理由
- 安全模块的变更是高风险操作，需要可追溯。
- 审计日志与命中日志分离，便于权限和 retention 管理。
